## 三、第二阶段：真实改进（按优先级）

### P0 · 图片预处理增强

> 优先级最高。OCR 质量是整个 Q1 的天花板，预处理直接决定 OCR 输入质量。
> 当前环境已有 OpenCV (`cv2`)，可以直接使用。

#### P0-1：偏斜校正（Skew Correction）

**目标：** 纠正拍照/扫描时产生的文档倾斜（±15° 以内）。

**实现方案：**
```
输入图片 → 灰度化 → Canny 边缘检测 → 霍夫变换(HoughLines)找直线
       → 计算所有直线的平均倾角 → 仿射变换旋转校正 → 输出
```

**参考代码骨架：**
```python
def _deskew_image(image: Image.Image, max_angle: float = 15.0) -> Image.Image:
    """Detect and correct document skew using Hough line detection."""
    import cv2
    import numpy as np

    gray = np.array(image.convert("L"))
    edges = cv2.Canny(gray, 50, 150, apertureSize=3)
    lines = cv2.HoughLines(edges, 1, np.pi / 180, 200)

    if lines is None:
        return image  # no clear lines, skip

    angles: list[float] = []
    for line in lines:
        rho, theta = line[0]
        angle = np.rad2deg(theta) - 90
        angle = max(-max_angle, min(max_angle, angle))
        angles.append(angle)

    if not angles:
        return image

    median_angle = np.median(angles)
    if abs(median_angle) < 0.5:
        return image  # negligible skew

    # Rotate
    h, w = gray.shape
    center = (w // 2, h // 2)
    M = cv2.getRotationMatrix2D(center, median_angle, 1.0)
    rotated = cv2.warpAffine(gray, M, (w, h), borderMode=cv2.BORDER_REPLICATE)
    return Image.fromarray(rotated)
```

**预期收益：** 扫描件 OCR 字符准确率 +2-4pp。FUNSD 表单字段空间对齐改善后，layout-based 提取也会受益。

#### P0-2：自适应二值化（Adaptive Thresholding）

**目标：** 解决当前全局阈值对光照不均图片效果差的问题。

**当前问题：**
```python
# 当前 strong 模式（preprocessor.py:343）
threshold = int(max(90, min(180, mean)))
work = work.point(lambda p: 255 if p > threshold else 0)
```
全局阈值在图片一侧暗一侧亮时，暗侧文字会全部变黑。

**改进方案：**
```python
def _adaptive_binarize(image: Image.Image, block_size: int = 11, c: int = 2) -> Image.Image:
    """Apply adaptive Gaussian thresholding for uneven lighting."""
    import cv2
    import numpy as np

    gray = np.array(image.convert("L"))
    # blockSize must be odd
    if block_size % 2 == 0:
        block_size += 1
    binary = cv2.adaptiveThreshold(
        gray, 255,
        cv2.ADAPTIVE_THRESH_GAUSSIAN_C,
        cv2.THRESH_BINARY,
        block_size, c,
    )
    return Image.fromarray(binary)
```

**预期收益：** 低质量扫描件 OCR 字符准确率 +5-8pp。

#### P0-3：非局部均值去噪（NL-Means Denoising）

**目标：** 去除噪点同时保留文字边缘，替代当前效果一般的 MedianFilter。

**当前问题：**
```python
# 当前 medium/strong 模式
work = work.filter(ImageFilter.MedianFilter(size=3))
work = work.filter(ImageFilter.UnsharpMask(radius=1.5, percent=140, threshold=3))
```
MedianFilter 会模糊文字边缘，UnsharpMask 又强化了噪点。

**改进方案：**
```python
def _denoise_nl_means(image: Image.Image, h: float = 10.0) -> Image.Image:
    """Non-local means denoising — preserves edges while removing noise."""
    import cv2
    import numpy as np

    gray = np.array(image.convert("L"))
    denoised = cv2.fastNlMeansDenoising(gray, None, h, 7, 21)
    return Image.fromarray(denoised)
```

**预期收益：** 噪点多的老文档/传真件 OCR 准确率 +3-5pp。

#### P0-4：文档边界检测（Document Boundary Detection）

**目标：** 对手机拍摄的文档照片，自动检测文档边界并透视校正。

**实现方案：**
```
灰度化 → 高斯模糊 → Canny边缘 → 轮廓检测 → 找最大四边形
      → approxPolyDP 逼近 → 四点透视变换 → 裁剪输出
```

**预期收益：** 手机拍照场景新增能力。配合偏斜校正后整体拍照文档可用性大幅提升。

#### P0-5：预处理策略自适应

**目标：** 替代当前固定的 light/medium/strong 三档策略。

**方案：** 在预处理前对图片做质量评估，根据评估结果自动选择处理链：

```
图片 → 质量评估
      ├─ 标准差 > 60（清晰扫描件）→ 仅 auto-contrast + 轻度锐化
      ├─ 标准差 35-60（普通扫描件）→ 偏斜校正 + auto-contrast + 锐化
      ├─ 标准差 20-35（低质量/传真）→ 偏斜校正 + 去噪 + 自适应二值化
      └─ 标准差 < 20 或边缘检测到四边形（拍照）
           → 文档边界检测 + 透视校正 + 偏斜校正 + 去噪 + 自适应二值化
```

### P1 · PDF 表格结构化提取

#### P1-1：表格与正文分离存储

**当前问题：** 表格直接拼入 LLM prompt 文本（[extractor.py:766-767](agent_file_create/document/extractor.py#L766-L767)）：
```python
if table_content:
    combined.append("--- 检测到的表格结构 ---\n" + table_content)
```
LLM 做语义总结时会丢失行列精度。

**改进方案：**
- 表格不进入正文 LLM prompt
- 单独存储为 `_structured_tables` 字段
- 每张表的内部结构：

```python
{
    "page": 3,
    "table_index": 1,
    "caption": "销售数据汇总",    # 从表上方/下方的文本推断
    "headers": ["月份", "销售额", "利润"],
    "rows": [
        ["1月", "120,000", "15,000"],
        ["2月", "98,000", "12,000"],
    ],
    "dimensions": {"rows": 2, "cols": 3},
    "markdown": "| 月份 | 销售额 | 利润 |\n| --- | ...",
}
```

- 正文 LLM prompt 中只引用表的摘要："第 3 页检测到一个 2 行 3 列的表格（表头：月份/销售额/利润）"
- 下游任务需要时可按 index 精确引用

#### P1-2：表格结构校验

提取后增加基本校验：
1. **列数一致性：** 每行列数相同，不一致时标记 + 尝试修复
2. **空表头检测：** 第一行全空的表可能不是表（是列表或格式残骸），加标记
3. **单元格截断检测：** OCR 区域宽度远大于提取出的文本长度 → 可能截断

#### P1-3：跨页表格合并

- 连续两页中都检测到无表头的表 → 可能是上一页的延续
- 按列数匹配 → 相同列数 → 合并 rows
- 合并后标记 `"continued_from": previous_page`

### P2 · OCR 质量提升

#### P2-1：LLM 辅助 OCR 纠错

**目标：** 替代硬编码映射表，用通用的 LLM 上下文推理来纠正 OCR 错误。

**方案：** 在 `_extract_image` 的 OCR + vision LLM 流程中，增加一个专门的 OCR 纠错步骤：

```
OCR 输出 → 低质量片段标记 → LLM prompt："以下是 OCR 可能含错的文本。
         请根据图片视觉信息和文档整体上下文，修正字段名和值的错误。"
         → 返回修正后的文本 → 进入后续提取流程
```

**标记规则（不需要硬编码字典）：**
- 连续 8+ 字符无空格 → 可能是粘连（"LORILLARPRLS"）
- 特殊字符比例 > 15% → 可能是乱码
- 置信度分数 < 0.7（如果 OCR 引擎提供）
- 包含少见的大写/小写混合模式（"MaECT", "Periars"）

#### P2-2：EasyOCR 字符集约束

**当前：** EasyOCR 不限制输出字符集，可能产生控制字符等乱码。

**改进：** 传入 `allowlist` 参数限制可输出字符：
```python
# 英文表单场景
reader = easyocr.Reader(['en'], gpu=False)
results = reader.readtext(image, allowlist="ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789 .,:/-()$#@&")
```

中文场景使用更宽的字符集。

### P3 · 表单/文档结构化理解

#### P3-1：表单字段类型分类

在布局提取 key-value 对之后，对每个 field 进行分类：

| 类型 | 检测特征 | 校验规则 |
|:----|:--------|:--------|
| DATE | 正则 `\d{1,2}[/\-\.]\d{1,2}[/\-\.]\d{2,4}` 或月份名 | 月 1-12，日 1-31 |
| PHONE | 正则 `[\d\s\-\(\)\.]{7,}` | 含 ≥7 位数字 |
| AMOUNT | `$` `€` `¥` `£` `元` 前缀，或纯数字+小数 | 数字格式 |
| ADDRESS | 含 Street/Road/Ave/路/街 + 数字 | 多行 |
| EMAIL | `@` + 域名 | — |
| NAME | 首字母大写 + 空格 + 首字母大写，或中文 2-4 字 | 不含数字（排除金额/电话） |

类型校验可以帮助发现 OCR 错误——例如日期字段 OCR 出来 `"2Dec7"` → 类型已知是 DATE → LLM 可修正为 `"Dec 2, 2007"`。

#### P3-2：表单区域分组

对于字段数 > 10 的表单，按空间位置自动分组：
- **表头/元数据区**（顶部）：日期、编号、收发件人
- **主题/内容区**（中部）：业务字段
- **签章/审批区**（底部）：签名、日期、批准

分组后传给 LLM，让 LLM 理解层次结构而非扁平 kv 列表，提升对复杂表单的理解。

### 4.2 评测指标

| 任务 | 当前指标 | 问题 | 新指标 | 说明 |
|:----|:------|:-----|:-----|:-----|
| 表格提取 | — | TabFact 测的是文本总结 | **TEDS** (Tree-Edit-Distance-Based Similarity) | 比较提取的表格 HTML 树与 ground truth 的结构相似度。业界标准。范围 [0, 1] |
| 表单提取 | key_value_match_rate (contains) | key/value 飘在文本里就算匹配 | **KV Precision / Recall / F1** | 精确比对 (field, value) 对：field 归一化 + value 归一化 → 完全匹配才算对 |
| 文档理解 | answer_hit_rate (contains) | 太弱，答案字符串出现就算对 | **ANLS** (Average Normalized Levenshtein Similarity) | 计算回答与标准答案的编辑距离。1 − 编辑距离/max(len(a), len(b))。范围 [0, 1] |
| 中文 OCR | 未实现 | — | **CER** (Character Error Rate) + **行准确率** | CER = 编辑距离字符数 / 总字符数。行准确率 = 完全正确的行 / 总行数 |

**辅助指标：**

- **字段填充率**（已有）：`filled_fields / total_fields`，反映信息完整度
- **Schema 合规率**（已有）：`required_ok` 为 True 的比例
- **提取耗时**：平均每个文件的处理时间（s）

### 4.3 评测脚本设计

文件：`scripts/eval_extract_q1_v2.py`

**设计原则：**

1. **结构化比对代替 contains**
2. **消融测试模式**：`--ablate skew|binarize|denoise|llm_fix` 逐一开关改进项
3. **模型上限校准**：`--model-ceiling gpt-4v` 用强模型跑同一样本集
4. **最少 50 条样本**：默认 `--sample 100`
5. **报告置信区间**：输出 `mean ± std`，例如 `TEDS: 0.72 ± 0.08`

**命令行接口：**

```
python scripts/eval_extract_q1_v2.py \
    --datasets pubtabnet,funsd,sroie,docvqa,ctw \
    --sample 100 \
    --text-model qwen3.5:9b \
    --vision-model minicpm-v:8b \
    --ablate "" \           # 空=全部开启；"skew,binarize"=关闭偏斜校正和二值化
    --model-ceiling "" \    # 可选强模型做上限校准
    --output result/q1_eval_v2_baseline.json
```

**输出格式：**


## 六、现实可达目标

### 6.1 与工作文档原目标对比

| 指标 | 数据集 | 估算当前真实基线 | 优化后现实目标 | 原工作文档目标 | 差异原因 |
|:-----|:------|:-----:|:-----:|:-----:|:-----|
| 表格 TEDS | PubTabNet | ~0.45-0.55 | **≥0.75** | ≥0.90 | 8B vision model 对复杂表格布局的理解有天花板 |
| 表单 KV F1 | FUNSD (50-100) | ~0.15-0.25 | **≥0.60** | ≥0.85 | OCR 是硬瓶颈，扫描件低分辨率字段无法完全恢复 |
| 收据 KV F1 | SROIE (100) | ~0.30-0.40 | **≥0.70** | — | 新增评测，发票/收据结构相对规整 |
| 文档 ANLS | DocVQA (100) | ~0.40-0.50 | **≥0.65** | ≥0.85 | GPT-4V 在 DocVQA 的 ANLS 约 0.83-0.88，8B 不可能超越 |
| 中文 OCR CER | CTW (200) | 待测 | **≤0.15** (即 ≥85% 准确率) | +5pp | 此目标合理 |

### 6.2 目标设定逻辑

1. **模型上限校准。** 用强模型（GPT-4V / Claude Vision）跑同一样本集，得到该测试集的可达上限。本地 8B 模型目标 = 上限 × 0.80-0.85。

2. **OCR 天花板。** FUNSD 表单中许多字段是手写体或超低分辨率印刷体，OCR 引擎本身无法识别。这类样本的分数天花板由 OCR 引擎决定，与预处理/后处理无关。KV F1 的合理上限约 0.70-0.75。

3. **不做数字游戏。** 优化目标是"在给定模型约束下最大化提取质量"，而非追求某个绝对数字。如果 PubTabNet TEDS 从 0.50 提升到 0.72，即使没到 0.75 也是显著进步。

## 七、关键原则

### 7.1 不做的事

- ❌ 不在代码中嵌入任何测试集特有的字段名、值、OCR 修正规则
- ❌ 不为了拉高分数向输出中堆砌文本
- ❌ 不修改评测脚本以放宽评测标准
- ❌ 不以 3 条样本的结果宣称任何结论

### 7.2 必须做的事

- ✅ 每个改进项必须有消融测试数据支撑
- ✅ 评测脚本和评测结果必须可复现（固定 seed + 记录模型版本）
- ✅ 最少 50 条样本才做结论
- ✅ 记录每次改进的"成本"（额外耗时、token 消耗）

### 7.3 整体节奏

```
第 1 周：阶段 1（清洗+基线）+ 阶段 2（图片预处理）
第 2 周：阶段 3（表格结构化）+ 阶段 4（OCR+LLM 协同）
第 3 周（可选）：如果指标未达预期，追加一轮 OCR 引擎升级（PaddleOCR）+ 表单 refinement
```
