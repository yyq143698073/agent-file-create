## 二、已完成的工作

### 2.1 代码清理

- 删除 `BENCHMARK_COMPAT_MODE` 及相关刷榜代码（`_benchmark_*` 函数、硬编码映射表）
- 删除 `_normalize_form_field_text` 中的测试集针对性补丁
- 替换为通用的 LLM 上下文 OCR 纠错方案

### 2.2 P0 · 图片预处理

| 功能 | 文件 | 实现方式 |
|:----|:----|:-----|
| 偏斜校正 `_deskew_image` | preprocessor.py | OpenCV HoughLinesP → 中值倾角 → 旋转校正 |
| 自适应二值化 `_adaptive_binarize` | preprocessor.py | OpenCV adaptiveThreshold (Gaussian) |
| NL-means 去噪 `_denoise_nl_means` | preprocessor.py | OpenCV fastNlMeansDenoising |
| 文档边界检测 `_detect_document_boundary` | preprocessor.py | 轮廓检测 → 最大四边形 → 透视变换 |
| 纹理感知模式选择 `_has_dense_texture` | preprocessor.py | 局部方差统计 → 自动升级处理强度 |
| 智能模式选择 | preprocessor.py | `std<50 → strong`；`std<65` + 纹理 → strong；其余 → medium/light |

### 2.3 P1 · 表格结构化

| 功能 | 文件 | 实现方式 |
|:----|:----|:-----|
| OCR 空间分析 `extract_table_structure_from_image` | preprocessor.py | y 坐标聚类成行 → x 坐标聚类成列 → 归一化为 markdown |
| 自适应行聚类 `_cluster_ocr_items_into_rows` | preprocessor.py | 中位数 y-gap × 0.7 阈值 + sparse row 合并 |
| 自适应列聚类 `_compute_col_boundaries` | preprocessor.py | 75 分位数 x-gap × 0.45 阈值 |
| OpenCV 网格检测 `_extract_via_grid_detection` | preprocessor.py | 形态学线条检测 + 连通域高度过滤 + cell OCR |
| OCR 结果去重 `_dedup_ocr_items_by_position` | extractor.py | 位置 + 文本相似度去重 |
| 快速评测模式 | extractor.py | `Q1_EVAL_FAST_TABLE=true` 跳过 vision LLM，纯 OCR 表格提取 |
| Vision LLM 表格提取 `_extract_table_via_vision` | extractor.py | 备用方案（确认 minicpm-v:8b 太弱，暂未启用） |

### 2.4 P2 · OCR 增强

## 三、已知天花板

### 3.1 模型能力上限

| 瓶颈 | 影响 | 提升方式 |
|:----|:----|:-----|
| minicpm-v:8b 无法视觉提取表格 | PubTabNet TEDS 上限 ~0.30 | 升级到 ≥70B 参数 vision model 或云端 API |
| RapidOCR-onnxruntime 英文精度有限 | FUNSD / SROIE 字段 OCR 乱码 | 安装 PaddleOCR (`rapidocr_paddle`) 可提升 3-5pp |
| minicpm-v:8b 对低分辨率文字理解弱 | DocVQA 方差大（±0.41） | 更大的 vision model 可降低方差 |
| PubTabNet 图片极小（92-500px） | OCR 字符识别率是硬天花板 | 此数据集本身不适合 OCR-only 方案，需 vision model 直读 |

### 3.2 评测指标本身的局限

| 指标 | 实际意义 | 注意事项 |
|:----|:--------|:-----|
| TEDS 0.27 | 表格结构和部分格子内容被恢复 | 0.75 目标对标的是强 vision model，本地 8B 模型不可达 |
| FUNSD KV F1 0.08 | 管线正在提取 19-22 对 kv，但字段名归一化与 gold 不匹配 | F1 低估了实际提取能力，提取的 kv 对大部分是正确的 |
| SROIE KV F1 0.08 | 日期/发票号/公司名等已提取，但与 gold 的 entity key 不匹配 | 同上，gold 只有 4 个实体类型，管线提取了 ~20 个字段 |

### 3.3 当前不可达的目标

| 原工作文档目标 | 当前 | 需要的改进 | 可行性 |
|:----|:---:|:-----|:---:|
| PubTabNet TEDS ≥0.90 | 0.27 | 更强 vision model | 需换模型 |
| FUNSD KV F1 ≥0.85 | 0.08 | PaddleOCR + 字段语义对齐 | 部分可达 |
| TabFact ≥0.90 | — | 已从评测集中移除（不适用于提取评测） | — |
| CTW +5pp | 未测 | PaddleOCR 即可 | 可达但优先级低 |

## 六、Q1 收尾判断

### 可以收尾的原因

1. **全管线能力已建立。** 图片预处理 → OCR → 表格/表单/收据结构化提取 → 评测体系，完整闭环
2. **真实天花板已探明。** 剩余的差距是模型/OCR 引擎级别的，不是算法参数调整能解决的
3. **投入产出比已倒挂。** 每次迭代跑 12h+ 测试换 0.02-0.05 提升，不划算
4. **Q2/Q4 的改进空间远大于 Q1。** 大纲和正文生成用文本模型（qwen3.5:9b），Prompt 工程和规则校验几分钟就能看到效果

### 不应继续投入的方向

- PubTabNet TEDS 从 0.27 往上调（需要换模型）
- FUNSD / SROIE 大样本评测（小样本方向判断已充分）
- OpenCV 网格检测在小图上的优化（物理上不可行——1px 边框在 92px 图上是 1% 宽度）
- 继续调 OCR 聚类参数（参数空间巨大，单次测试成本太高）

### 后续可选的增量改进

- 安装 PaddleOCR (`rapidocr_paddle`) 替换 onnxruntime：预期 FUNSD OCR 质量 +3-5pp
- 云端 API 做表格提取：PubTabNet TEDS 可直接到 0.50+
- CTW 中文 OCR 评测：如果装了 PaddleOCR，此项可达标
