### 1.3 评测方法

**与传统 Q1 评测不同，Q3 评测不需要 LLM 调用**，纯算法级别：

```
BEIR 文档 → 分块入库 → 对每个标注查询执行检索 → 计算 top-K 命中标注文档的比例
```

评测脚本：（新建）`scripts/eval_retrieval_q3.py`

```
python scripts/eval_retrieval_q3.py \
    --datasets nfcorpus,scidocs,fiqa,msmarco \
    --sample 200 \
    --top-k 10 \
    --ablate "" \
    --output result/q3_eval_v2_baseline.json
```

**评测指标：**
- **Recall@10**：top-10 命中标注相关文档的比例
- **Precision@10**：top-10 中标注相关文档数 / 10
- **MRR@10**：第一个相关段落的倒数排名的均值
- **跨 chunk 截断率**：随机采样 100 条检索结果段落，人工判定是否语义断裂

## 二、优化项

### 2.1 P0 · 自适应 RRF k 值

**当前状态：** RRF 的 k 值固定为 60，三路融合（向量/BM25/词法）权重相同。

**问题：** 不同类型的查询需要不同的融合策略：
- 含数字/代码的查询 → 词法匹配更重要（k 值应偏小，让词法排名主导）
- 长句语义查询 → 向量匹配更重要（k 值应偏大，让向量排名主导）
- 短关键词查询 → RRF 均匀融合即可

**优化方案：**

在 `store.py` 的 RRF 融合逻辑中，根据查询特征动态调整 k 值：

```python
def _adaptive_rrf_k(query: str) -> float:
    """Return an adaptive RRF k based on query characteristics."""
    digits_ratio = sum(1 for c in query if c.isdigit()) / max(len(query), 1)
    code_chars = sum(1 for c in query if c in '{([<>!=+*&|^~%]})')
    query_len = len(query)
    
    if digits_ratio > 0.15 or code_chars >= 3:
        return 30   # lexical-heavy query → smaller k biases toward lexical
    elif query_len > 40:
        return 90   # long semantic query → larger k biases toward vector
    else:
        return 60   # default
```

**同时按查询特征调整三路权重：**

```python
def _adaptive_rrf_weights(query: str) -> tuple[float, float, float]:
    """Return (vector_weight, bm25_weight, lexical_weight) for RRF."""
    digits = sum(1 for c in query if c.isdigit())
    code_chars = sum(1 for c in query if c in '{([<>!=+*&|^~%]})')
    long_query = len(query) > 50
    
    if digits / max(len(query), 1) > 0.1 or code_chars >= 2:
        return (0.6, 0.3, 0.1)  # lexical favored
    elif long_query:
        return (0.3, 0.3, 0.4)  # balanced with vector bias
    else:
        return (0.4, 0.3, 0.3)  # default balanced
```

**文件：** `agent_file_create/rag/store.py`  
**影响范围：** `search()`, `search_adaptive()` 的 RRF 融合逻辑  
**预期收益：** Recall@10 +3-5pp（通过让不同类型查询用不同的融合策略）

### 2.2 P1 · 查询扩展增强

**当前状态：** 查询扩展已有 jieba 分词 + 品牌剥离 + 评测后缀 + 缩写展开。但从知识点的扩展仅用 `_generate_queries_from_knowledge_point`。

**优化：** 让查询扩展同时利用**大纲正文**（不仅标题）和**已生成的上文**。

#### 2.2.1 从大纲正文提取术语

当前 `plan_section_knowledge` 构建查询列表时：
```python
queries.extend(_extract_title_concepts(section_title))  # 仅标题
queries.extend(_extract_title_concepts(parent_title))    # 父标题
```

改为：
```python
queries.extend(_extract_title_concepts(section_title))
queries.extend(_extract_title_concepts(parent_title))
# 新增：从大纲正文中提取关键词
if section_body:  # 如果大纲已经生成了章节描述
    queries.extend(_extract_key_terms_from_body(section_body, max_terms=5))
```

**函数 `_extract_key_terms_from_body`：** 用 jieba TF-IDF 或 TextRank 从章节描述文本中提取 top-5 关键词作为额外查询。

#### 2.2.2 上文感知查询扩展

当生成到第 N 个 H2 章节时，前面 N-1 个章节的生成结果已经可用。利用这些上文来增强查询：

```python
# 对第 3+ 个章节，从已生成的相邻章节提取领域术语
if section_index >= 2 and previous_sections:
    prev_body = "\n".join(
        sec.get("generated_content", "")[:500] 
        for sec in previous_sections[-2:]
    )
    queries.extend(_extract_key_terms_from_body(prev_body, max_terms=3))
```

**文件：** `agent_file_create/rag/planner.py`  
**预期收益：** Recall@10 +2-3pp（更精准的查询词覆盖）

### 2.3 P2 · MISSING 知识点定向追加检索

**当前状态：** Planner 有三层质量反馈循环（`_quality_ok()` 检查），但不满足时只是切换到下一层策略，**不做定向追加检索**。

**问题：** 如果一个知识点完全没有匹配到任何内容（所有 hit 分数 < 0.3），说明当前查询策略不合适，应该用**不同角度**的查询再试一次。

**优化方案：**

在 `_retrieve_by_section_type` 的质量反馈循环末尾，增加 MISSING 检测：

```python
def _retrieve_with_missing_fallback(kb, queries, section_type, ...):
    hits = _retrieve_by_section_type(kb, queries, section_type, ...)
    
    # 检测是否存在 MISSING 知识点（覆盖率为 0 的知识点）
    missing_kps = _detect_missing_knowledge_points(queries, hits, threshold=0.30)
    
    if missing_kps:

        alt_queries = _generate_alternative_queries(missing_kps)
        extra_hits = kb.search_adaptive(alt_queries, top_k=8)
        hits = _merge_hits(hits, extra_hits)
    
    return hits


def _detect_missing_knowledge_points(queries, hits, threshold=0.30):
    """找出覆盖率低于阈值的知识点"""
```

### 2.4 P3 · 内容去重增强

**当前状态：** 已有文档级多样性交织。但可能存在**内容相似但 chunk ID 不同**的重复段落。

**优化方案：**

在 `plan_section_knowledge` 的 hit 收集阶段，增加内容哈希去重：

```python
def _dedup_by_content_hash(hits: list[Hit]) -> list[Hit]:
    """对相似内容（编辑距离 < 阈值）的 hit 仅保留最高分的一个"""
    seen_hashes: set[str] = set()
    deduped: list[Hit] = []
    
    for hit in sorted(hits, key=lambda h: h.score, reverse=True):
        # 使用归一化后的内容指纹（去空格、去标点、小写）
        fingerprint = _content_fingerprint(hit.content)
        if fingerprint not in seen_hashes:
            seen_hashes.add(fingerprint)
            deduped.append(hit)
    
    return deduped


def _content_fingerprint(text: str, n: int = 50) -> str:
    """生成内容指纹——取最长的 n-gram 哈希"""
    normalized = re.sub(r"\s+", "", str(text or "").strip().lower())
    # 取中部最有代表性的片段
    start = max(0, len(normalized) // 3)
    snippet = normalized[start:start + n * 3]
    return hashlib.md5(snippet.encode()).hexdigest()
```

**文件：** `agent_file_create/rag/planner.py`  
**预期收益：** Precision@10 +5-8pp（减少重复内容浪费 top-K 位置）

### 2.5 P4 · 跨 chunk 截断检测

**当前状态：** 无截断检测。chunk 可能在中文字/英文字中间截断。

**优化方案：**

在分块时增加截断边界检查，优先在语义完整边界处分块：

```python
def _find_safe_split_point(text: str, target_pos: int, window: int = 50) -> int:
    """在目标位置附近找到安全的切割点（句子边界）"""
    # 向后搜索句子边界
    for i in range(target_pos, min(len(text), target_pos + window)):
        if text[i] in "。！？\n" or (text[i:i+2] in {". ", "! ", "? "}):
            return i + 1
    # 向前搜索
    for i in range(target_pos, max(0, target_pos - window), -1):
        if text[i] in "。！？\n" or (text[i:i+2] in {". ", "! ", "? "}):
            return i + 1
    return target_pos  # 找不到，返回原位
```

同时在评测阶段增加截断率统计。

**文件：** `agent_file_create/rag/chunker.py`  
**预期收益：** 跨 chunk 截断率降低 5-8pp

## 五、与 Q1 的经验教训

1. **先建评测再优化。** Q1 的问题是改了代码再测，不知道每个改动的独立贡献。Q3 应该先在 BEIR 上跑基线，再做消融。

2. **小样本快迭代。** Q3 的评测不需要 LLM，5 条查询 2 秒跑完。用 5-10 条快速验证方向，200 条做最终结论。

3. **设置现实目标。** Q1 原目标 0.85 对标的是 GPT-4V 级别。Q3 的目标应基于实测基线设定——先跑基线，再定目标。

4. **记录天花板。** Q1 最终文档记录了已知天花板，Q3 也应如此：embedding 模型 `bge-m3` 的 Recall 上限是多少？RRF 的理论最优 k 值范围？这些决定了优化的上限。
